# Devlog 002 ‚Äî M1: the protocol package, and what a two-stage review caught

**2026-07-06**

Milestone M1 is done: `packages/protocol` now holds the full HomeFleet Protocol v0.1.0 surface as zod schemas ‚Äî node identity & capabilities, the job model (recon + command as a discriminated union), five streaming event types, pairing, errors, and the RPC wrappers ‚Äî with the RFC draft at `docs/rfc/hfp-v0.md` kept field-for-field in sync. 92 tests, TDD throughout.

## Process note: the review loop earned its keep

The workflow here is implementer ‚Üí spec-compliance review ‚Üí code-quality review, each with fresh eyes. The spec review passed clean. The quality review came back with **14 findings** ‚Äî zero correctness bugs, but exactly the kind of foundation cracks that get expensive once other packages depend on the inferred types:

- **Type-level lies.** `JobResult.status` was runtime-refined to terminal states but *typed* as all five ‚Äî so every downstream `switch` would have needed dead branches for `"queued"`/`"running"` states that can never occur. Fixed with a real `TerminalJobStatusSchema`.
- **Two sources of truth inside the source-of-truth package.** The terminal-status list existed twice (a refine and a private predicate). Now: one exported `TERMINAL_JOB_STATUSES` + type-guard.
- **A genuine zod v4 pitfall:** `schema.default(value)` *short-circuits* ‚Äî the default value bypasses parsing. Our recon budgets restated the field defaults inside `.default({...})`, so changing a field default would silently desync. `prefault({})` parses the empty object *through* the schema, so field defaults stay the single source of truth. Worth remembering.
- **Validating structure but not consistency.** A `JobSnapshot` could claim `status: "succeeded"` while carrying a result saying `"failed"` ‚Äî structurally valid, semantically contradictory. Cross-field `superRefine`s now reject every such mismatch (snapshot‚Üîresult status and jobId, event‚Üîresult jobId, error-presence rules: failed *requires* an error, succeeded *forbids* one).

Plus a batch of small tightenings: canonical-lowercase IDs everywhere, semver-validated version fields, integer constraints on byte counts, 1-hour caps on both wall-time budgets, `z.json()` instead of `z.unknown()` for wire-safe error details, boundary tests at every limit, and an exported `HFP_PATH_PREFIX` so the transport layer can't drift from the protocol version.

One deliberate late change: `JobResult` gained a required `type` discriminator. The MCP front (M6) will emit results as structured output, and consumers shouldn't have to infer the result shape from which optional fields happen to be present. This was the last free window for a breaking change ‚Äî the RFC is a draft and nothing speaks the wire format yet. From M2 onward, the versioning rules apply to us too.

## Takeaway

None of the 14 findings was a bug *today*. All of them were interest payments scheduled for M2‚ÄìM6. A protocol package is the one place where "works now" is the wrong bar ‚Äî the bar is "can't be misused later."
