# Devlog 007 — M6: the front door, where an empty list means "off"

**2026-07-07**

Milestone M6 is done: HomeFleet has a front door. The daemon now exposes an MCP server with five tools — `list_nodes`, `delegate_task`, `job_status`, `job_result`, `cancel_job` — so whatever MCP client the user already runs (Claude Code, LM Studio, goose…) becomes the cockpit. This is ADR-0002 made real: we ship no UI; the calling agent's UI is the UI. 375 tests green.

The MCP front is the *delegating* side — its tools drive the M5 `HfpClient` to talk to remote worker nodes; it runs no jobs itself. `list_nodes` joins the trust store (who we're paired with) against discovery (where they are) and does a best-effort `hello` for live capabilities. `delegate_task` resolves a node, calls `delegate`, and records the returned jobId against that node in a bounded registry so the bare-jobId tools (`job_status`/`job_result`/`cancel_job`) know where to route. Tool outputs reuse the protocol's `JobResult`/`JobStatus` zod schemas verbatim, so the agent-facing surface can't drift from the wire contract.

## Verify the SDK, don't remember it

The MCP SDK (`@modelcontextprotocol/sdk` v1.29.x) is HomeFleet's first external runtime dependency, and its API is specific and evolving. Rather than code against a remembered shape, the implementer read the installed types: the stateless Streamable HTTP pattern (`sessionIdGenerator: undefined`, a fresh `McpServer` + transport per request), `registerTool` with zod input *and* output schemas returning `structuredContent`, and the DNS-rebinding options — which, notably, the SDK marks `@deprecated` in favor of external middleware but which remain the functional built-in mechanism in v1.29.x. Verifying beat assuming; the deprecation tag would have sent a guess down the wrong path.

## The security trap review caught: fail-open by omission

The daemon's front door binds to `127.0.0.1` with Origin validation and DNS-rebinding protection — the boundary that stops a malicious web page from driving your daemon via a local fetch. It was implemented correctly, and it was tested (a foreign Host and a foreign Origin each get a 403 against the real transport). But the review found the boundary was robust *by luck, not by construction*.

The allow-lists (`allowedHosts`/`allowedOrigins`) can't be built until the ephemeral port is known, which is after `listen`. So they started as `[]` and were filled in once the port bound. The trap: the SDK's check is `if (allowedHosts.length > 0)` — **an empty allow-list disables validation entirely.** The code was safe only because the fill-in ran in the microtask continuation before any incoming request (a macrotask) could be serviced. That's real, but it's incidental to event-loop ordering: one future `await` inserted before the assignment, and the daemon's security boundary silently fails *open*. The fix makes it structural — build the lists synchronously inside the `listen` callback (the port is available there) before the server can accept anything, and add a defensive guard that refuses with 503 if the lists are ever empty. A security boundary should fail closed when something is wrong, never open.

## The other asymmetry: an unbounded body on the hardened daemon

The HFP node server has enforced `MAX_BODY_BYTES` since M2 — a deliberate cap so a peer can't balloon memory with a giant request. The MCP HTTP path had quietly skipped it, handing the request straight to the SDK, which reads the whole body with no limit. Loopback-only binding means the caller is a local process, so the blast radius is small — but the asymmetry with the hardened path is exactly the kind of gap that becomes a real one later. The fix owns the body read: cap by content-length up front, cap the streamed bytes for the chunked/no-content-length case, and hand the SDK the already-parsed body (verified against the SDK source to short-circuit its own read, so there's no two-consumers race). Malformed JSON now gets a clean 400 at the door, too.

## Takeaway

Every milestone so far has had one lesson about untrusted input; M6's is subtler than the others because the front door was already *mostly* right. The finding wasn't a missing check — it was a correct check that depended on timing to stay correct. "Tested and passing" told us the boundary worked today; it took reading *why* it worked to see it was one refactor away from silently not working. Fail-closed-by-construction is the difference between a boundary that holds and one that happens to hold.

The stdio shim rounds out the milestone: a thin bin that reuses the exact same tool registration over stdio, for clients that can't speak Streamable HTTP. Pointing Claude Code at it and delegating a real job is the first thing to try on the rig — that live test is the human's to run.

Next (M7): workspace sync — git-bundle transfer of a repo to the worker (full bundle first, incremental after), unbundled into a per-repo cached workspace, with a worker-side repo allowlist. That's what turns `delegate_task`'s `WorkspaceRef` from a promise into an actual checkout the executor can read.
