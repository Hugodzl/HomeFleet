# Devlog 006 — M5: dispatch end-to-end, and the streams that must not hang

**2026-07-07**

Milestone M5 is done, and it's the one where HomeFleet first does the thing it exists to do: a daemon accepts a delegated job, runs it through an executor, streams the events back, returns a result, and honors cancellation — proven across N daemons on one machine over real loopback mTLS. 345 tests green. The pieces built over M1–M4 (protocol schemas, mTLS transport, the executors) finally connect through a job manager and four new HTTP routes.

The protocol package earned its keep here: `DelegateRequest`/`DelegateResponse`, `JobSnapshot`, `CancelResponse`, and the `JobEvent` union already existed from M1, with their cross-field superRefines. M5 wrote zero new wire schemas — it implemented the endpoints the RFC already described. That's the payoff of treating the protocol as the source of truth a milestone early.

## SSE without a hole in the auth wall

The events stream is Server-Sent Events, which meant teaching the transport's route layer — until now a JSON request/response affair — to hand a handler the raw response and stream on it. The temptation is to special-case SSE somewhere that skips the machinery, and the M2 review had explicitly warned against exactly that. So the streaming route is a new *kind* of route that runs **after** the identical `identifyPeer → paired` chokepoint every JSON route passes; the auth gate doesn't know or care that what follows is a stream. Unpaired peers get the same 401 they'd get anywhere, and the job-ownership check (below) runs before a single SSE byte — before `writeHead(200)` — so an unauthorized request gets a plain JSON 404 and never learns whether the job exists.

## Ownership is the whole authorization model

A job is owned by the peer that submitted it, keyed by client-certificate fingerprint. Every accessor — snapshot, cancel, subscribe — goes through one `getOwned` gate, and the crucial property is that "not yours" and "doesn't exist" are *indistinguishable*: same `UNKNOWN_JOB` code, same status, synchronous map lookup so there's no timing tell. A paired peer C cannot probe which jobs peer A delegated to B. The integration suite drives a third daemon against another's job to prove all three accessors return `UNKNOWN_JOB`.

## The bounds, again

By now the refrain is familiar — every milestone that touches peer-driven input asks "what grows without limit?" M5's answers: the job queue is bounded (`BUSY` past the cap, no unbounded backlog), terminal-job retention is bounded (evict oldest-terminal, logged), and the client's SSE parser caps both per-event and total stream bytes against an untrusted server. Records are keyed only by server-minted UUIDs, never peer data.

## What review caught: two ways a stream hangs forever

The dispatch logic reviewed clean on correctness — exactly-once event delivery (a snapshot-then-register with no `await` between replaying the buffer and adding the subscriber), no double-terminal, the running slot released on every exit path. The findings were about *liveness*, and both are the kind that only bite on a real network:

- **A dead server hangs the client.** The streaming client had set `socket.setTimeout(0)` — no inactivity timer for the life of the stream. If the server's machine sleeps or is unplugged (no clean FIN), `for await` never sees an end and the iterator blocks forever. The server sends a 15-second heartbeat, but a heartbeat only helps if the client arms a timer that the heartbeats *reset*. It now does: a 45-second idle timeout (3× the heartbeat) that a live stream keeps alive and a silent peer trips, raising the same `HfpTimeoutError` the unary path uses.
- **A stuck executor hangs the daemon.** `cancel()` and `stop()` awaited the executor's unwind unconditionally. The M4 executors self-bound (the spawn kill-grace, the agent's fetch-abort), so it's fine today — but M6 brings *new* executors, and one that ignored its abort signal would wedge the cancel HTTP handler and block daemon shutdown. So both now race the unwind against a bounded timeout: the happy path is unchanged (well-behaved executors always win, and cancel still returns the true terminal status), but a misbehaving one gets reported honestly rather than hanging the process.

Both fixes came with the test that would have caught them — a stream whose server goes silent, and a `StuckExecutor` that ignores cancellation — plus a no-network `JobManager` unit suite locking in the engine paths (retention eviction, a throwing executor, post-terminal event drops, concurrent cancel, `stop()` closing live subscribers) that the integration tests only exercised indirectly.

## Takeaway

The dispatch *correctness* was never really in doubt once the event-ordering invariants were pinned. The risk was all in the failure modes you can't see on loopback: the peer that stops responding without saying goodbye. On one machine every socket closes cleanly; on two machines across a house, they don't. M5's hardening is really M8 insurance bought early, while the code is fresh.
