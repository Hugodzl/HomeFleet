# Devlog 014 — v0.2 rig smoke: a real code-writing delegation on hardware

**2026-07-15**

The v0.2 write path had been proven only by the in-process E2E test (two daemons,
one machine, mocked model). Today it ran for real: the **laptop** (delegator)
delegated a scoped `write` task to the **tower** (worker) running
`qwen3.6-35b-a3b` on llama.cpp/Vulkan, the local model wrote the change, and the
laptop fetch-applied it as `homefleet/897e3bebabac` into the source repo. It
worked on the first try, and the landed test passes on the laptop.

## What ran

Task: "add a unit test for the config loader" (repoId `homefleet`). The tower's
Qwen read `config.ts` and neighbours (5 tool calls, ~39.5k prompt tokens), wrote
a new `packages/daemon/src/config/config.load-smoke.test.ts` (817 completion
tokens), and called `finish_task`. The daemon committed it as
`HomeFleet Worker <worker@263d9c76.invalid>`, bundled the branch, streamed it
back, and the laptop applied it under the reserved namespace.

**Verified on the laptop** (not taken on faith from `JobResult`): the branch tip
equals `artifact.headCommit` (`44956f84`); the author is the worker identity; the
diff is exactly the one new 15-line file; the working tree and `main` were
untouched; and the landed test runs green (`vitest` 1/1). `artifactStatus:
applied`.

## Numbers (full data in the rig `benchmarks.md`, "Job 4")

- delegate + incremental git-bundle sync: 5.3 s
- end-to-end (delegate → terminal): ~169 s; executor wall ~162 s
- apply (fetch + non-forced fetch into `refs/heads/homefleet/`): 0.83 s
- 5 tool calls; 39,501 prompt / 817 completion tokens; ~22 tok/s gen
  (tower-reported). The 817-tokens-over-162 s aggregate is a floor, not the
  steady-state rate — it includes 5 tool round-trips and ~39.5k prompt tokens
  (KV-cache reuse across calls).

## What it proves, and what it doesn't

Proven on real heterogeneous hardware: the authority-moving path that mattered in
the design — a git-free executor producing a `HomeFleet Worker` commit, bundled
out, and landed by a **non-forced** fetch confined to `refs/heads/homefleet/`
without touching the user's tree or branches — end to end, with a real local
model doing the edit.

Honest limits: one small, well-scoped task whose target behaviour is
near-duplicated by an existing `config.test.ts` case (chosen for a reliable first
smoke, not novelty); no `verifyCommand` exercised on the worker (deferred — the
laptop ran the test post-apply instead); and a single 35B model. A harder,
multi-file task and the worker-side verify path remain untested on hardware.

Model quality was good: correct imports and assertions, no hallucinated APIs, a
genuinely passing test. One harmless nit — `mkdtemp(\`${tmpdir()}homefleet-smoke-\`)`
omits a path separator, so the temp dir lands one level above `tmpdir` and is
never cleaned up; the test still passes.

## Process notes

Reused the M8 two-session + LAN-mailbox pattern: the laptop posted a full brief
into the mailbox and drove coordination; a tower Claude session did the tower-side
bring-up (pull to v0.2, `pnpm build`, add the new `executors.write` block pointing
at llama-server, restart the daemon) and reported readiness. Three things worth
recording:

- The M8 Job-2 longpaths bug did **not** recur — v0.2's `workerConfig` pins
  `core.longpaths=true`. The fix held on the machine that first surfaced it.
- llama-server's OpenAI tool-calling was probed before the run
  (`finish_reason=tool_calls` parsed) — cheap insurance, since the write loop is
  useless without it.
- A concurrent laptop session advanced this shared checkout's `main` to `7487792`
  (a `.gitignore`-only commit) mid-smoke; the delegation naturally used the
  current HEAD, and per the shared-checkout rule this was verified, not
  reacted to. Also: the tower's Claude box was missing the
  `homefleet-rig-coordination` memory and reconstructed the launch from the M8
  devlog — worth syncing that memory to the tower.
